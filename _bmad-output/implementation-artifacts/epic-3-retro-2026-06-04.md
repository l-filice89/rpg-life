## What Went Well

### Domain-first sequencing prevented drift
Story 3.1 established `@rpg-life/domain` with pure functions and test vectors before complete/reward work. That made 3.2 and 3.5 rule enforcement consistent and auditable.

### Completion flow shipped with strong safety controls
Story 3.2 landed transaction boundaries, idempotent retry behavior, owner scoping, race protections, and schema validation for timezone inputs.

### Reward UX and accessibility quality improved through review
Story 3.3 started functional, then tightened a11y labels, reduced-motion behavior, hydration behavior, and duplicate continue-flow calls through focused patches.

### Profile became a trustworthy progression surface
Story 3.4 now always returns all seven skills, keeps ordering stable, and uses RSC fetch patterns with loading and error boundaries so progression is visible and consistent.

### Focus rules are now end-to-end enforceable
Story 3.5 closed the Epic 2 stubs with server-side spend transactions, conflict handling, discriminated input validation, and user-neutral copy across spend actions.

### Empty-state experience now reflects user context correctly
Story 3.6 cleanly separates first-time empty vs board-clear empty and ties board-clear display to complete/reward flow without conflating the two UX states.

---

## Challenges and Growth Areas

### Review patch volume clustered around interactive client flows
The heaviest patch rounds appeared in modal orchestration, mutation error handling, and in-flight guards. These are recurring complexity hotspots for this codebase.

### Date validation is still fragmented
Several stories deferred full calendar-valid date parsing. Regex checks exist, but invalid calendar dates are still a known class requiring shared validation utilities.

### Duplicate utility logic is growing
`isOverdue`-style checks and adjacent helpers are duplicated across service/router/util layers, increasing the risk of subtle behavior divergence.

### Some deferred a11y/robustness concerns remain
Custom level-up overlay focus trapping and one icon-path hard-fail case were deferred as acceptable for MVP scope but should not be carried indefinitely.

## Key Insights

1. **Pure domain core pays compounding dividends:** once progression rules were centralized, later stories moved faster and avoided rule drift.
2. **Mutation UX needs a default guardrail checklist:** pending state, retry copy, invalidation targets, and modal transition rules should be mandatory scaffolding.
3. **Board-state transitions are product logic, not just UI polish:** first-empty vs board-clear distinction carries behavioral meaning and must stay explicit in code.
4. **Code review is catching high-impact issues pre-merge:** concurrency, a11y, and data integrity concerns were found before becoming runtime regressions.
5. **Deferred validation/util consolidation is now a real maintenance risk:** shared date and overdue helpers should be elevated before more flow complexity lands.
